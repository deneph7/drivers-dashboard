> **최종 업데이트**: 2026-03-11 (인터뷰 기반 상세화)

## 1. 작업 컨텍스트

### 1.1 배경 및 목적

조직 내 운전기사(최대 10명)의 실시간 가용 상태를 관리자/비서가 한눈에 파악할 수 있는 웹 대시보드 서비스를 구축한다. 기사는 본인의 상태를 직접 입력하고, 관리자는 대시보드에서 전체 현황을 모니터링하며 필요 시 상태를 대리 변경할 수 있다.

### 1.2 범위

| 포함 | 제외 |
|------|------|
| 기사 상태 입력/변경 | GPS 기반 자동 위치 추적 |
| 실시간 대시보드 | 운행 경로/이력 관리 |
| 사용자 인증 및 권한 관리 | 급여/정산 시스템 |
| 차량번호 매핑 | 차량 정비/관리 |
| 반응형 모바일 대응 | 네이티브 앱 |

### 1.3 입출력 정의

**입력:**
- 기사: 가용 상태(파란/빨간/회색), 차량번호, 복귀 예정시간(빨간불 시)
- 관리자: 사용자 계정 생성/관리, 기사 상태 대리 변경

**출력:**
- 실시간 대시보드 화면 (신호등 UI, 복귀 예정시간, 차량번호, 현재 날짜/시간)
- 복귀 지연 경고 표시

### 1.4 제약조건

- 비용: 무료 인프라 범위 내 운영 (Supabase Free + Vercel Free)
- 사용자 규모: 기사 최대 10명 + 관리자/비서 소수
- 데이터 영속성: 세션 무관하게 상태 유지 (DB 저장 필수)
- 보안: 로그인 인증 필수 (기사, 관리자 모두)
- 반응형: 모바일(핸드폰) ~ 데스크톱까지 대응

### 1.5 용어 정의

| 용어 | 설명 |
|------|------|
| **가용(파란불)** | 기사가 현재 업무 투입 가능한 상태 |
| **불가용(빨간불)** | 외근/운행 중으로 투입 불가. 복귀 예정시간 필수 입력 |
| **연락필요(회색불)** | 가용/불가용 여부가 불명확한 상태. 관리자가 직접 연락해서 확인해야 함. 기사가 스스로 설정하는 '모름' 상태 |
| **복귀 지연** | 빨간불 상태에서 복귀 예정시간이 경과했으나 상태 미변경 |
| **이니셜** | 기사를 식별하는 짧은 표기. 형식 자유 (한글 성씨, 영문 이니셜, 숫자 등 관리자가 직접 설정) |

### 1.6 기술 스택 결정

| 영역 | 선택 | 근거 |
|------|------|------|
| **프레임워크** | Next.js (App Router) | 풀스택 단일 프로젝트, SSR/API 라우트 통합 |
| **DB + 인증** | Supabase (PostgreSQL + Auth) | 무료 티어 충분 (500MB DB, 50K MAU), 실시간 구독 지원 |
| **배포** | Vercel | Next.js 네이티브 지원, 무료 티어 |
| **실시간 업데이트** | Supabase Realtime | DB 변경 시 대시보드 자동 반영, 추가 비용 없음 |
| **스타일링** | Tailwind CSS | Next.js와 기본 통합, 반응형 유틸리티 |

## 2. 워크플로우 정의

### 2.1 사용자 역할 및 권한

| 역할 | 권한 |
|------|------|
| **관리자(admin)** | 대시보드 조회, 기사 상태 대리 변경 (대시보드 카드 클릭 → 모달), 사용자 계정 관리(생성/삭제/비밀번호 초기화), 기사 프로필 편집(이니셜, 이름, sort_order), 사이트 언어 설정 |
| **기사(driver)** | 대시보드 조회, 본인 상태 변경(가용/불가용/연락필요), 차량번호 입력, 복귀 예정시간 입력 |
| **비서(viewer)** | 대시보드 조회 전용 (상태 변경 불가, 로그인 필수) |

> **권한 요약 매트릭스:**
>
> | 역할 | 대시보드 조회 | 본인 상태 변경 | 전체 상태 변경 | 사용자 관리 | 정렬 순서 편집 |
> |------|-------------|--------------|--------------|-----------|--------------|
> | admin | ✅ | ✅ | ✅ | ✅ | ✅ |
> | driver | ✅ | ✅ | ❌ | ❌ | ❌ |
> | viewer | ✅ | ❌ | ❌ | ❌ | ❌ |

### 2.2 핵심 워크플로우

#### 워크플로우 A: 기사 상태 변경

> **UX 원칙: 모바일 속도 최우선** — 상태 버튼 탭 즉시 저장 (확인 단계 없음). 빨간불은 복귀시간 입력이 필수이므로 예외적으로 2단계.

```
[기사 로그인] → /dashboard (대시보드)
    │
    ▼
[대시보드에서 본인 카드 클릭]
    │
    ▼
[상태변경 모달(StatusModal) 표시 — 슬라이드업 애니메이션]
    │
    ├─ 파란불(가용) 탭
    │   └─ 차량번호 입력 (선택, recent_vehicles 자동완성)
    │       └─ [저장] → DB Upsert → 모달 닫힘 → 대시보드 실시간 반영
    │
    ├─ 빨간불(불가용) 탭
    │   └─ 복귀 예정 날짜+시간 입력 (필수, 현재시간 이후만 허용)
    │       └─ 차량번호 입력 (선택, recent_vehicles 자동완성)
    │           └─ [저장] → DB Upsert → 모달 닫힘
    │               └─ [복귀 예정시간 경과 시] 대시보드에 '복귀 지연' 경고 표시
    │
    └─ 회색불(연락필요) 탭
        └─ [저장] → DB Upsert → 모달 닫힘
```

**분기 조건:**
- 빨간불 선택 시 → 복귀 예정시간 입력 필드 활성화 (필수, 날짜+시간, 현재시간 이전 불가)
- 파란불/회색불 선택 시 → 복귀 예정시간 필드 비활성화/초기화

**차량번호 자동완성:**
- `profiles.recent_vehicles text[]`에 저장된 최근 5개 번호 드롭다운 표시
- 저장 성공 시 배열 앞에 신규 번호 추가, 6번째 항목 이후 제거

#### 워크플로우 B: 관리자 대시보드 조회 및 대리 변경

```
[관리자 로그인] → /dashboard
    │
    ▼
[대시보드 화면]
    │
    ├─ 전체 기사 상태 실시간 조회 (신호등 UI)
    │   ├─ 각 기사: 이니셜, 상태, 차량번호, 복귀 예정시간
    │   ├─ 복귀 지연 기사: 경고 배지 표시
    │   └─ 현재 날짜/시간 표시 (KST 기준)
    │
    └─ 특정 기사 카드 클릭 (관리자만 → 모든 카드 클릭 가능)
        └─ 상태변경 모달(StatusModal) 표시 — 슬라이드업 애니메이션
            └─ 워크플로우 A와 동일한 StatusSelector 재사용
                └─ 저장 → DB Upsert (updated_by = admin.id 기록) → 모달 닫힘 → 대시보드 실시간 반영
```

> **설계 판단 (변경)**: 별도 편집 페이지(`/admin/driver/[id]`)에서 **대시보드 위 슬라이드업 모달**로 변경. 이유: (1) 페이지 전환 없이 즉시 상태 변경 가능, (2) StatusSelector 컴포넌트 재사용 유지, (3) driver도 대시보드에서 본인 카드 클릭으로 동일한 UX 제공.

#### 워크플로우 C: 관리자 사용자 관리

```
[관리자 화면 > 사용자 관리]
    │
    ├─ [최초 배포 시] 시드 스크립트 실행
    │   └─ 환경변수(SEED_ADMIN_EMAIL, SEED_ADMIN_PASSWORD)로 최초 admin 계정 생성
    │
    ├─ 신규 사용자 추가
    │   └─ 이름, 이메일, 역할(기사/비서/관리자), 이니셜, sort_order 입력
    │       └─ Supabase Auth로 계정 생성 → 초기 비밀번호 이메일 발송
    │           └─ driver 역할인 경우 → driver_status 레코드 자동 생성 (status='contact')
    │
    ├─ 기존 사용자 편집
    │   └─ 이니셜, 이름, 역할, sort_order 변경
    │
    ├─ 비밀번호 초기화
    │   └─ Supabase Auth 비밀번호 리셋 이메일 → 기사 본인 이메일로 발송
    │
    ├─ 사용자 비활성화/삭제
    │
    └─ 사이트 언어 설정
        └─ 영어(기본) / 한국어 토글 → localStorage 또는 site_settings 저장
```

#### 워크플로우 D: 복귀 지연 감지 (클라이언트 사이드)

```
[대시보드 렌더링 시 / 매 1분 주기]
    │
    ▼
[빨간불 상태인 기사 목록 조회]
    │
    ▼
[각 기사의 복귀 예정시간 vs 현재시간 비교]
    │
    ├─ 현재시간 < 복귀 예정시간 → 남은 시간 카운트다운 표시
    └─ 현재시간 ≥ 복귀 예정시간 → '복귀 지연' 경고 배지 표시
```

> **설계 판단**: 복귀 지연 감지는 서버 크론잡이 아닌 **클라이언트 사이드 타이머**로 처리한다. 사유: (1) 상태 자동 전환이 아니라 경고 표시만 하면 되므로 DB 업데이트 불필요, (2) Vercel 무료 티어에서 크론잡 제한 회피, (3) 대시보드가 열려있을 때만 의미 있는 정보.

### 2.3 LLM 판단 영역 vs 코드 처리 영역

> **결론: 이 프로젝트는 LLM 런타임 판단이 불필요하다.**

| 처리 영역 | 해당 작업 | 방식 |
|-----------|----------|------|
| **전량 코드 처리** | 인증/인가, 상태 CRUD, 실시간 구독, UI 렌더링, 시간 비교/경고 | 결정론적 로직 |
| **LLM 불필요 근거** | 모든 분기 조건이 명시적(상태값 3종, 시간 비교), 자연어 생성/분류/추론 없음 | — |

> 에이전트 시스템은 **구현 자동화**(Claude Code가 코드를 생성)에만 사용되며, 런타임에서 LLM을 호출하지 않는다.

### 2.4 단계별 성공 기준 및 검증

| 단계 | 성공 기준 | 검증 방법 | 실패 시 처리 |
|------|----------|----------|-------------|
| **인증** | 로그인 성공 시 역할(admin/driver/viewer)에 맞는 화면 라우팅 | 규칙 기반: 역할별 접근 권한 테스트 | 자동 재시도(최대 3회) → 에러 메시지 |
| **상태 변경** | DB에 상태값 저장, 빨간불 시 복귀시간 필수 존재 | 스키마 검증: status enum, return_time NOT NULL (빨간불) | 자동 재시도(최대 2회) → 사용자에게 오류 표시 |
| **실시간 반영** | 상태 변경 후 3초 이내 대시보드 갱신 | 규칙 기반: Supabase Realtime 구독 동작 확인 | 스킵 + 로그 (폴백: 30초 주기 폴링) |
| **복귀 지연 감지** | 예정시간 경과 시 1분 이내 경고 표시 | 규칙 기반: 타이머 주기(1분) 내 감지 | 스킵 + 로그 (다음 주기에 재감지) |
| **반응형 UI** | 360px~1920px 해상도에서 레이아웃 깨짐 없음 | 사람 검토: 주요 디바이스 크기 수동 확인 | 에스컬레이션 → 사람이 CSS 수정 지시 |
| **사용자 관리** | 계정 생성/삭제/비밀번호 초기화 정상 동작 | 규칙 기반: CRUD 기능 테스트 | 자동 재시도(최대 2회) → 에러 메시지 |
| **최종 산출물** | 전체 화면 동작, 역할별 접근 제어, 데이터 영속성 | 사람 검토: 전체 시나리오 워크스루 | 에스컬레이션 → 수정 지시 |

### 3.6 DB 스키마 개요

**테이블: `profiles`** (Supabase Auth 확장)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid (PK, FK → auth.users) | 사용자 ID |
| name | text | 이름 |
| initial | varchar(10) | 이니셜/표시명 (자유 형식: 한글, 영문, 숫자 등) |
| role | enum('admin', 'driver', 'viewer') | 역할 |
| recent_vehicles | text[] | 최근 사용 차량번호 목록 (최대 5개, FIFO) |
| sort_order | integer | 대시보드 카드 정렬 순서 (NULL 시 이름 알파벳순) |
| created_at | timestamptz | 생성일 |

**테이블: `driver_status`** (기사당 1개 레코드, Upsert 패턴)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| driver_id | uuid (PK, FK → profiles.id) | 기사 ID (UNIQUE — 기사당 1개 레코드) |
| status | enum('available', 'unavailable', 'contact') | 가용/불가용/연락필요 |
| vehicle_number | varchar(20) | 차량번호 (nullable, 매 상태 변경 시 갱신) |
| return_time | timestamptz | 복귀 예정 날짜+시간 — UTC 저장, 클라이언트에서 KST(UTC+9) 표시. 불가용 시 필수 |
| updated_at | timestamptz | 최종 변경 시각 (UTC) |
| updated_by | uuid (FK → profiles.id) | 변경한 사람 (본인 또는 관리자 — UI 미표시, DB만 기록) |

> **RLS(Row Level Security) 정책:**
> - `driver_status`: 인증된 사용자 전체 조회(SELECT) 가능 / driver는 본인 레코드만 UPDATE 가능 / admin은 전체 UPDATE 가능 / viewer는 읽기 전용
> - `profiles`: 인증된 사용자 전체 조회 가능 / admin만 INSERT·UPDATE·DELETE 가능
>
> **타임존 처리:**
> - DB 저장: UTC (PostgreSQL `timestamptz`)
> - 클라이언트 표시: `Intl.DateTimeFormat('ko-KR', { timeZone: 'Asia/Seoul' })` 또는 `date-fns-tz`로 KST 변환
>
> **신규 driver 계정 생성 시:** `driver_status` 레코드 자동 생성 (`status='contact'`, 나머지 NULL)

### 3.7 화면 구성 개요

#### 화면 A: 로그인 (`/login`)
- 이메일 + 비밀번호 입력 폼
- 로그인 성공 시 모든 역할 → `/dashboard`
- 로그인 중 "Signing in..." 로딩 오버레이 표시

#### 화면 B: 대시보드 (`/dashboard`)
- 접근 가능 역할: admin, driver, viewer (모두)
- **헤더**: 현재 날짜 및 실시간 시계 — KST 기준 (초 단위 갱신)
- **기사 카드 그리드**: `sort_order ASC` 정렬 (NULL인 경우 이름 알파벳순), 반응형 (모바일 1열, 태블릿 2열, 데스크톱 3열)
  - 각 카드 구성:
    - 이니셜 (큰 글씨)
    - 신호등 아이콘 (파란/빨간/회색 원형)
    - 차량번호 (있을 경우)
    - 빨간불: 복귀 예정시간(KST) + 남은 시간 카운트다운
    - 복귀 지연 시: 빨간 깜빡임 또는 경고 배지
  - **admin 접속 시**: 모든 카드 클릭 가능 → StatusModal 모달로 상태 대리 변경
  - **driver 접속 시**: 본인 카드만 클릭 가능 → StatusModal 모달로 상태 변경
  - **viewer**: 카드 클릭 불가 (조회만)
  - 클릭 가능한 카드에는 편집 아이콘 표시
- **범례**: 신호등 색상 의미 안내

#### 화면 C: 상태 변경 모달 (StatusModal — 대시보드 위 오버레이)
- 대시보드에서 카드 클릭 시 슬라이드업 모달로 표시
- driver: 본인 카드만 클릭 가능 / admin: 모든 카드 클릭 가능
- 현재 상태 표시 (신호등)
- 3개 버튼: 가용 / 불가용 / 연락필요
- 불가용 선택 시: 복귀 예정 날짜+시간 입력 (datetime picker, 현재시간 이전 불가)
- 차량번호 입력 필드 (`VehicleInput.tsx` — `recent_vehicles` 기반 자동완성)
- 저장 성공 → 모달 닫힘 → Realtime으로 대시보드 자동 반영
- ESC 키 또는 배경 클릭으로 모달 닫기

#### 화면 D: 관리자 화면 (`/admin`)
- 사용자 목록 테이블 (이름, 이메일, 역할, 이니셜, sort_order)
- 신규 사용자 추가 폼
- 사용자별 편집/비밀번호 초기화/비활성화 액션
- `sort_order` 직접 숫자 편집으로 카드 정렬 순서 조정
- 사이트 언어 토글 (영어 기본 / 한국어)

#### 화면 E: 기사 대리 변경 (레거시 — `/admin/driver/[id]`)
- **현재는 대시보드 모달(StatusModal)로 대체됨**
- 레거시 페이지는 호환성을 위해 유지하지만, 주요 UX 흐름에서 제외

### 3.10 타임존 및 국제화(i18n) 패턴

**타임존:**
- DB에는 항상 UTC로 저장 (`timestamptz`)
- 모든 시간 표시는 클라이언트에서 KST(Asia/Seoul, UTC+9)로 변환
- 사용 라이브러리: `date-fns-tz` 또는 브라우저 내장 `Intl.DateTimeFormat`
- 서버 사이드 렌더링 시: `Intl.DateTimeFormat(..., { timeZone: 'Asia/Seoul' })` 사용

**언어(i18n):**
- 사이트 전체 단일 언어 (사용자별 선택 없음)
- 기본값: **영어(English)**
- 관리자가 `/admin` 화면에서 한/영 전환 가능
- 언어 설정 저장: `localStorage['site_language']` (서버 불필요, 클라이언트 전용)
- 번역 문자열: `/src/lib/i18n.ts` — `{ en: {...}, ko: {...} }` 구조

## 4. 비기능 요구사항

### 4.1 보안
- Supabase RLS로 DB 수준 접근 제어:
  - `driver_status`: 전체 인증 사용자 SELECT / driver 본인 레코드 UPDATE / admin 전체 UPDATE
  - `profiles`: 전체 인증 사용자 SELECT / admin만 INSERT·UPDATE·DELETE
- Next.js 미들웨어로 라우트 보호:
  - `/admin/*` → admin 역할만 접근
  - `/driver` → driver 역할만 접근 (레거시, 주요 흐름은 대시보드 모달)
  - `/dashboard` → 모든 인증 사용자 접근
  - 로그인 시 모든 역할 → `/dashboard` 리다이렉트
- 비밀번호는 Supabase Auth가 해시 처리 (bcrypt)
- API 라우트에서 세션 토큰 검증 필수
- 최초 admin 계정: 시드 스크립트 (`/scripts/seed-admin.ts`) — 환경변수로 이메일/비밀번호 주입, 배포 후 1회 실행

### 4.2 성능
- Supabase Realtime으로 상태 변경 즉시 반영 (목표: 3초 이내)
- 폴백: Realtime 연결 실패 시 30초 주기 폴링
- 클라이언트 타이머: 1분 주기로 복귀 지연 체크

### 4.3 가용성
- Vercel: 99.99% SLA (무료 티어)
- Supabase: 무료 티어 — 7일 미활동 시 프로젝트 일시정지 가능 → **GitHub Actions cron job** (5일 주기 자동 ping)으로 방지

## 6. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| Supabase 무료 티어 7일 비활성 일시정지 | 서비스 중단 | **GitHub Actions cron job** (5일 주기)으로 자동 ping — 완전 자동화 |
| Supabase 무료 티어 용량 초과 | 서비스 중단 | 사용자 10명 이하이므로 가능성 극히 낮음. 모니터링 |
| Supabase Realtime 연결 불안정 | 대시보드 업데이트 지연 | 폴링 폴백 구현 (30초 주기) |
| 기사가 상태 업데이트를 잊음 | 대시보드 정보 부정확 | 회색불 상태의 존재 이유. 관리자가 대시보드에서 카드 클릭 → 모달로 대리 변경 가능 |
| 동시 변경 충돌 (기사+관리자) | 데이터 불일치 | Last-write-wins + `updated_at` 기록. 소규모라 발생 가능성 낮음 |
| 기사가 어제 빨간불 상태로 오늘 로그인 | 잘못된 상태 표시 | 의도적 설계 (DB 상태 유지). 기사가 직접 갱신하거나 관리자가 대리 변경 |
